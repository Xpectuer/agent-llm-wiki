# Code Review

> Source(s): raw/ch-02-Code Review Best Practices

Code review 是对源代码变更在合并入主分支前的系统性检查。它的多重目标包括：发现缺陷、共享知识、保持代码一致性以及指导初级开发者。来自 SmartBear 的研究表明，评审者每个 session 可以有效审查 200-400 行代码，超过这一阈值后缺陷发现率显著下降。这一发现意味着，**小且聚焦的 pull request 更易审查，并能带来更高质量的反馈**。

## 主要目标

代码审查的首要目标远不止缺陷检测。**知识共享**可能是最有价值的产出：当多位开发者都理解某段代码时，项目面对团队变动会更具韧性。代码审查也是保持代码库一致性的闸门，确保新代码遵循已有模式、命名约定和架构决策。一致性能够降低所有在代码库中工作者的认知负荷。

## 最佳实践

有效的代码审查需要技术与人际双重技能。评审者应区分**阻塞问题**与**建议**：

- **阻塞问题**：包括正确性缺陷、安全漏洞和架构违规，这些问题会在生产环境引发实际故障。
- **建议**：涵盖风格偏好、替代方案和可处理但不关键的小改进。

使用“nit:”或“suggestion:”等标签有助于传达反馈的严重性。[Conventional Comments](https://conventionalcomments.org/) 提供了代码审查反馈的结构化格式，帮助审查者清晰表达意图。

**审查速度**对团队节奏很重要，但不应牺牲彻底性。一个好的实践是在一个工作日内审查 pull request。若无法快速完成全面审查，可先留下初始确认消息，让作者知道其工作已进入队列。一些团队根据 `CODEOWNERS` 文件或轮询自动分配审查者，确保负载均衡和领域专长覆盖。为了帮助审查者跟踪队列中的任务，团队可以使用 Slack 通知、邮件摘要或仪表盘工具来避免遗漏。

## 自动化辅助

自动化检查应处理机械性问题，让人工审查者聚焦于设计与逻辑：

- **Linters** 强制风格规则
- **格式化工具**确保代码外观一致
- **类型检查器**捕捉类型错误
- **静态分析器**识别潜在缺陷

pre-commit 钩子在代码到达审查前本地运行这些检查，减少琐碎修正的来回沟通。持续集成（CI）在每次推送时运行全套检查与测试，让审查者对代码满足基本质量标准有信心。工具如 linters、formatters、type checkers 和 static analyzers 的自动化序列能显著提升效率。

## 健康的文化

培育健康的审查文化需要刻意努力：

- **审查代码，而非作者**：优先使用“这段代码可以通过……改进”，而非“你在这里犯了错”。
- **作者应以开放心态面对审查**：认识到建议是关于代码质量，而非个人能力。
- **当分歧出现时**：团队可在实现前使用 RFC（Request for Comments）文档讨论设计决策，避免审查阶段出现大量返工。提前达成共识能让后续审查更顺畅。

## 替代机制

结对编程和集体编程提供了将审查前移的代码质量机制：

- **结对编程**：两个开发者共用一个工作站——一人写代码，另一人实时审查。这种持续审查能立即捕捉问题，消除异步审查的延迟。
- **集体编程**：扩展到整个团队共同解决同一问题。这些方法用个人灵活性换取更高的 bus factor 和实时知识共享。

## 审查清单

代码审查清单有助于审查者记住检查常见问题。清单条目可能包括：

- 是否有测试？
- 测试是否覆盖边界情况？
- 错误处理是否恰当？
- 是否有安全顾虑？
- 代码是否引入新的依赖？
- 文档是否更新？
- 此变更是否影响其他团队或服务？

使用清单降低了记住所有审查维度的认知负荷，并帮助初级审查者形成系统化的审查方法。
