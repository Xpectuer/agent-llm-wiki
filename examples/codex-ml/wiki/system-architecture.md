# 系统架构

> Source(s): raw/ch-03-Continuous Integration and Delivery, raw/ch-04-Design Patterns in Modern Software

系统架构是软件系统的基础骨架，它决定了系统的结构稳定性、演化能力和运行效率。架构决策贯穿整个开发生命周期，从技术选型、组件边界划分到通信协议和部署策略，每一层选择都会对后续的开发流程、质量保障和运维方式产生深远影响。

## 架构的核心维度

### 组件边界与职责

架构首先要划定组件的职责范围和相互依赖关系。传统的分层架构（如 MVC）将展示、业务逻辑和数据访问分离；而微服务架构则进一步将业务功能拆分为独立的服务单元。合理的边界划分可以降低耦合度，提升内聚性，并允许团队独立开发和部署。

设计模式在此过程中提供标准化的解决方案。例如，**工厂模式**（[[factory-pattern]]）和**策略模式**（[[strategy-pattern]]）将创建逻辑和算法变体从使用处解耦，使得架构能够灵活应对变化。**观察者模式**（[[observer-pattern]]）在事件驱动架构和消息系统中被广泛采用，用于实现组件间的松耦合通信。

### 通信模式与数据流

组件间的通信方式直接影响系统的响应性、可靠性和复杂度。同步调用（如 REST、gRPC）简单直接，但容易产生级联故障；异步消息（如事件总线、消息队列）能提升弹性，但增加了最终一致性和调试难度。选择通信模式时需权衡**性能**、**可扩展性**和**一致性要求**。

在分层架构中，数据流通常是单向的：用户请求经过控制器、服务层、数据访问层后返回响应。事件驱动架构则允许数据以事件的形式在各个服务间流动，每个服务可以独立订阅和发布事件，形成更灵活的数据管道。

### 部署拓扑与基础设施

部署拓扑决定了系统如何映射到物理或虚拟资源上。从单体部署到微服务容器化，再到无服务器架构，每种拓扑都有自己的成本模型和运维复杂度。

架构决策直接影响 [[continuous-delivery|持续交付]] 流程的实现方式。例如，采用微服务架构的团队通常需要更精细的 CI/CD 管道，每个服务独立构建、测试和部署。**流水线即代码**（[[pipeline-as-code]]）确保这些管道可版本化、可复审，实现跨仓库的一致实践。

## 架构对开发流程的影响

### 分支策略与集成频率

**主干开发**（[[trunk-based-development]]）是配合持续集成（CI）的典型分支策略：开发者在短命分支（通常不超过一天）上工作，频繁合并到主分支。这种策略要求架构支持特性开关（feature flags），让不完整的功能能安全地存在于生产代码中。相比之下，长期存在的功能分支会积累合并债务，延迟集成反馈。

架构的模块化程度直接影响分支策略的可行性：强内聚、松耦合的模块允许团队并行开发不同的功能，而不会产生大量冲突。

### 构建性能与开发效率

慢速的 CI 管道会造成上下文切换成本——开发者在等待构建结果时转向其他任务，返回时需重新进入状态。架构层面的决策（如服务拆分粒度、依赖关系复杂度）会影响构建性能：

- **依赖缓存**和**并行化测试**可加速管道。
- 使用构建产物（如 Docker 镜像）避免重复工作。
- 分式测试（distributed testing）在多台机器上并行执行独立测试套件。

一些组织维护专用的构建集群，为所有团队提供一致且快速的 CI 执行环境。

## 架构中的典型模式与权衡

### 单例模式与依赖管理

**单例模式**（[[singleton-pattern]]）确保类只有一个实例并提供全局访问点。虽然简单，但在架构层面容易产生隐藏依赖和测试困难。现代架构更倾向于使用依赖注入容器显式管理对象生命周期，在测试代码中提供测试专用实例，而不是依赖全局单例。

### 依赖管理和安全

架构的组件依赖关系图越复杂，依赖管理的挑战越大。过期的依赖可能包含安全漏洞，及时升级可减少断裂变更的积累。工具如 Dependabot 和 Renovate 自动化依赖更新，通过创建 Pull Request 触发完整 CI 管道来验证兼容性。锁文件（如 `package-lock.json`、`poetry.lock`、`Cargo.lock`）确保可重现构建。

### 安全性和成本

架构中的每个组件边界都是潜在的攻击面。微服务架构增加了网络通信的安全风险（如中间人攻击、API 暴露），需要额外的认证、授权和流量加密措施。同时，更多的服务意味着更高的运维成本（监控、日志、链路追踪）。架构师必须在**安全性**与**成本**之间做出权衡，例如在关键路径上使用更严格的安全措施，而非全量覆盖。

## 架构演进与持续优化

系统架构不是一成不变的，它会随着业务需求、技术环境和团队规模的变化而演化。通过以下实践保持架构的健康：

- **架构评审**：定期审查核心组件的耦合度、技术债务和性能瓶颈。
- **增量优化**：使用 [[neural-architecture-search|神经架构搜索]] 等自动化方法优化模型部署拓扑，对业务系统则可通过分析通信模式重构服务边界。
- **可观测性**：部署成熟的监控、日志和追踪系统，为架构决策提供数据支持。
