# CQRS（命令查询职责分离）

> Source(s): raw/ch-03-Continuous Integration and Delivery, raw/ch-04-Design Patterns in Modern Software

CQRS（Command Query Responsibility Segregation，命令查询职责分离）是一种架构模式，它将系统中的读操作（查询）和写操作（命令）分离到不同的模型、服务甚至存储中。这一模式源自 Bertrand Meyer 在《面向对象软件构造》中提出的“命令查询分离原则”（CQS），但 CQRS 将其扩展到整个架构层面：命令（Command）负责修改状态，不返回数据；查询（Query）负责返回数据，不修改状态。

## 核心理念

- **命令（Command）**：执行变更操作（创建、更新、删除），不返回业务数据，仅返回操作状态（成功/失败/错误）。命令通常以任务的形式表示，如 `PlaceOrderCommand`、`RegisterUserCommand`。命令在领域模型中执行，触发领域事件和状态变更。
- **查询（Query）**：读取数据，返回结果集，不修改状态。查询可以针对专门优化的读模型（如物化视图、缓存、搜索索引）进行，以提高查询性能和灵活性。

## 为什么需要 CQRS？

在传统 CRUD（Create-Read-Update-Delete）系统中，读和写共用同一个数据模型和存储。当系统复杂度增长时，这种统一模型会带来问题：
- **性能冲突**：写操作需要事务一致性和验证逻辑，读操作追求快速返回和灵活查询。两者对索引、缓存、数据组织的要求不同。
- **领域复杂性**：查询需求往往需要跨多个聚合的数据视图（如报表），而命令操作严格限制在聚合边界内。强制用同一模型应对两者会导致“大泥球”架构。
- **可扩展性**：读写负载差异大时（如社交网络读远多于写），统一模型难以独立扩展。CQRS 允许读模型和写模型分别部署和扩展。

## CQRS 与事件溯源（Event Sourcing）

CQRS 常与事件溯源（Event Sourcing）结合使用。事件溯源将状态变更存储为事件序列，而不是直接存储当前状态。写模型持久化事件，读模型从事件流中投影出视图。这种组合提供了完整的审计日志、时点回放能力，并允许从不同角度重建读模型。

## CQRS 的设计模式关联

CQRS 并非孤立模式，它与许多经典设计模式相互配合：
- **命令模式（Command Pattern）**：CQRS 中的 Command 可直接借鉴命令模式的设计，将请求封装为对象，支持参数化、队列化、日志化。
- **策略模式（Strategy Pattern）**：读模型可以策略化地选择不同的投影方式或缓存策略。
- **观察者模式（Observer Pattern）**：写模型发布领域事件，读模型作为观察者订阅并更新视图，形成事件驱动的数据流。
- **工厂模式（Factory Pattern）**：用于构建复杂的命令对象和查询对象，封装实例化逻辑。
- **仓库模式（Repository Pattern）**：在写模型侧，仓库用于持久化聚合；在读模型侧，可用自定义存储（如缓存仓库）优化查询。

## 实践中的考量

CQRS 不是银弹，适用于具有以下特征的场景：
- 读写负载严重不对称（如 IoT 设备数据写入与仪表盘查询）。
- 需要从不同角度展示同一数据（如 CRM 系统中的客户详情视图、客诉视图、活动历史视图）。
- 团队希望围绕业务能力进行微服务拆分，读写分离有助于明确边界。

实施 CQRS 时需注意复杂性增长：
- **最终一致性**：读模型通常异步更新，用户可能看到旧数据。需要权衡一致性和性能，并在 UI 层提供反馈（如“数据正在同步”）。
- **数据同步**：读模型必须可靠地从写模型获取变更。可使用消息队列、变更数据捕获（CDC）等技术。
- **重复逻辑**：读模型和写模型都需实现业务规则？不，业务规则由写模型（领域模型）强制执行，读模型只负责投影和展示——读模型通常不做验证，仅用写模型已确认的聚合数据生成视图。

## 与持续集成/持续交付（CI/CD）的关系

（材料中虽未直接提及 CQRS，但 CI/CD 实践对 CQRS 系统尤为重要）采用 CQRS 的系统架构更复杂，读模型和写模型可能作为独立服务部署。CI/CD 管道需要针对每个模型分别构建、测试和部署，并保证数据同步组件的健康。Pipeline as Code 可以定义多阶段发布策略，如先部署写模型，再部署读模型，并用特性开关控制新读写模型的渐进式上线。

## 相关模式与概念

- **命令查询分离原则（CQS）**：CQRS 的微观基础，强调方法级别不应同时修改状态和返回值。
- **事件溯源（Event Sourcing）**：存储状态变更序列，与 CQRS 天然互补。
- **领域驱动设计（DDD）**：CQRS 适用于复杂领域，常与聚合、领域事件配合。
- **读写分离（Read/Write Splitting）**：数据库层面的读写分离，但 CQRS 扩展到了模型层次的分离。
