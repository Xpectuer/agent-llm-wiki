# 工厂模式 (Factory Pattern)

> Source(s): raw/ch-03-Continuous Integration and Delivery

工厂模式是面向对象设计中的创建型模式，主要包括 **工厂方法模式 (Factory Method)** 和 **抽象工厂模式 (Abstract Factory)**。其核心思想是：将对象创建的逻辑从客户端代码中抽离出来，封装在专门的工厂类或工厂方法中，客户端通过调用工厂来获得所需实例，而不是直接使用 `new` 关键字构造具体类。

这种解耦带来了显著好处：

- **客户端与具体实现类分离**：即使具体类发生变化（如替换为新的子类），只要工厂接口不变，客户端代码无需修改。
- **创建逻辑集中管理**：可以在工厂中统一处理依赖注入、配置读取、环境判断等横切关注点。
- **运行时多态创建**：工厂可以根据配置、环境变量、参数等运行时条件，决定返回哪个具体子类的实例。

工厂模式在实际工程中最常见的应用场景包括：

- **插件架构**：系统通过工厂加载插件实现，客户端只依赖抽象的插件接口，具体插件由工厂根据注册信息动态生成。
- **跨平台抽象**：例如 GUI 工具包中的按钮工厂，在 Windows 平台返回 WinButton，在 macOS 平台返回 MacButton，客户端仅调用 `ButtonFactory.createButton()`。
- **测试替身**：在单元测试中，工厂可以返回 mock 对象或 stub，从而隔离外部依赖。

## 工厂方法 vs 抽象工厂

| 模式 | 目的 | 典型结构 |
|------|------|----------|
| **工厂方法 (Factory Method)** | 定义一个创建对象的接口，但让子类决定实例化哪个类。 | 一个抽象 Creator 类，其 `factoryMethod()` 由子类实现。 |
| **抽象工厂 (Abstract Factory)** | 提供一个创建一系列相关或相互依赖对象的接口，而无需指定具体类。 | 一个抽象工厂接口包含多个创建方法（如 `createA()`, `createB()`），每个具体工厂生产一族产品。 |

两者常组合使用：抽象工厂中的每个创建方法内部可以调用具体子类的工厂方法。

## 相关实践

在持续集成与持续交付（CI/CD）的上下文中，工厂模式常用于构建可配置的构建管道。例如，Pipeline as Code 中的 `actions`、`steps` 或 `tasks` 可以通过工厂模式根据参数动态生成（如 GitHub Actions 中自定义 action 的加载逻辑）。同样，在依赖注入容器中，对象的生命周期管理本质上是工厂模式的高级变体。

工厂模式也经常与策略模式 (Strategy Pattern) 配合：策略模式封装算法族，而工厂模式则负责根据运行时条件选择合适的策略实例。

## 与其他模式的关系

- **单例模式 (Singleton)**：工厂内部可维护一个单例实例，但工厂本身应避免成为全局单例，建议通过依赖注入注入工厂。
- **模板方法模式**：工厂方法本身是模板方法的一种应用——基类定义创建步骤，子类提供实现。
- **建造者模式 (Builder)**：当对象构造过程复杂且需要多个步骤时，建造者模式比工厂更合适；工厂通常适用于单一方法创建对象。
