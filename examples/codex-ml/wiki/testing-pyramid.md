# 测试金字塔 (Testing Pyramid)

> Source(s): raw/ch-01-Testing Strategies and Methodologies

测试金字塔是由 Mike Cohn 推广的软件测试分层策略模型，旨在指导团队合理分配不同类型测试的比例，从而在覆盖率、执行速度与维护成本之间取得平衡。该模型建议项目应包含大量单元测试（底层）、适量集成测试（中层）和少量端到端测试（顶层）。

## 单元测试

单元测试专注于应用程序中最小的可测试部分，通常是单个函数或方法。优秀的单元测试应具备快速、隔离和确定性。常用框架包括 pytest（Python）、JUnit（Java）和 Jest（JavaScript）。Arrange-Act-Assert（AAA）模式是常见结构：先设置测试数据，再执行被测函数，最后验证输出是否符合预期。Mock 对象和 Stub 用于隔离外部依赖（如数据库、网络调用、文件系统）。然而，过度模拟会导致测试变得脆弱，验证的是实现细节而非行为，增加重构难度。

## 集成测试

集成测试验证不同模块或服务是否协同工作。单元测试确认单个组件独立运行正常，而集成测试能捕捉接口不匹配、数据格式问题和通信协议错误。数据库集成测试验证查询在真实数据库实例上返回正确结果；API 集成测试检查端点是否正确处理身份验证、请求验证和响应格式化。Testcontainers 等库提供可丢弃的 Docker 容器，便于在测试执行期间快速启动真实的数据库、消息队列等服务。

## 端到端测试

端到端测试模拟完整的用户工作流程，例如用户注册、登录、创建内容再退出。常用工具包括 Selenium、Playwright 和 Cypress（用于 Web 应用自动化）。这些测试提供了系统按预期工作的最高置信度，但也是速度最慢、最脆弱的。由于定时问题、网络变化和浏览器渲染差异，端到端测试中容易产生 Flaky 测试（间歇性通过或失败）。缓解策略包括使用显式等待、重试机制以及在确定性环境中运行测试。

## 其他测试类型

- **属性基测试**：高级方法，测试指定应适用于任何有效输入的通用属性，框架自动生成随机输入来验证。与基于示例的测试不同，属性基测试能发现开发人员可能未想到的边界情况。Python 的 Hypothesis 库提供与 pytest 集成的属性基测试能力。
- **测试驱动开发 (TDD)**：先写测试再写实现代码。红绿重构循环：先写失败的测试，再写最小代码使其通过，最后改进代码结构并保持测试绿色。TDD 支持者认为它通过让测试成为代码的第一个客户端来强制思考接口和可用性，从而产生更好的设计。批评者认为 TDD 会减慢初始开发速度，可能不适合探索性或研究型项目。
- **回归测试**：确保新变更不会破坏已有功能。持续集成流水线通常在每次拉取请求时运行完整测试套件，在部署前捕获回归。前端开发中流行的快照测试会捕获组件渲染输出并与保存的快照比较，以检测意外的视觉变化。
- **性能测试**：评估系统在负载下的行为。负载测试测量预期流量下的响应时间和吞吐量；压力测试将系统推向超过正常容量以寻找断裂点。常用工具包括 JMeter、Locust 和 k6。使用 Python 的 timeit 模块或 C++ 的 Google Benchmark 进行微基准测试有助于识别性能回归。性能测试应在隔离环境中运行以确保结果一致。
- **可访问性测试**：确保应用能被残障人士使用。axe-core 和 Lighthouse 等工具根据 WCAG 指南审计网页。
- **安全测试**：识别 SQL 注入、跨站脚本和身份验证缺陷等漏洞。静态应用安全测试 (SAST) 工具分析源代码，动态应用安全测试 (DAST) 探测正在运行的应用。两者在现代 DevSecOps 流水线中不可或缺。
