# ralph-wiggum-loop

> Source(s): raw/Harness engineering leveraging Codex in an agent-first world.md

## 概述

**ralph-wiggum-loop** 是一种代码审查循环策略，其中代理（如 Codex）在本地和云端自行审查代码，迭代直到满足所有审查条件，无需人类介入。该术语源自开发团队中工程师通过代理运行代码审查的实践，旨在将人类从重复的审查工作中解放出来，使工程团队能够专注于更高层次的设计和系统构建。

该策略最初由 Ryan Lopopolo 在 Harness 团队的工程实践中提出，并在其博文 “Harness engineering: leveraging Codex in an agent-first world” 中详细描述。团队在构建一个包含约一百万行代码的产品时，采用了 0 行手动编写代码的约束，并依赖 Codex 代理完成所有代码生成、审查和迭代工作。

## 核心机制

ralph-wiggum-loop 的核心是让代理自主完成代码审查的生命周期：

1. **本地审查**：工程师通过提示描述任务，运行代理生成代码并打开 pull request（PR）。代理首先在本地审查自己的更改，确保代码质量、风格和逻辑正确性。
2. **云端审查**：代理将本地更改推送到云端，在云端环境中请求额外的代理审查。这些审查包括对代码结构、测试覆盖、文档完整性等方面的评估。
3. **迭代循环**：代理响应任何人类或代理提出的反馈，进行修改并重新提交，直到所有代理审查者满意。这个循环完全由代理驱动，无需人类介入审查过程。
4. **最终合并**：一旦循环结束（所有审查条件满足），PR 可以自动合并。人类工程师可以选择是否审查最终结果，但在实践中，大多数审查工作已由代理间自动完成。

## 关键设计原则

### 环境规范化

为了支持代理自主审查，团队必须确保环境高度规范化和可重现。关键实践包括：

- 应用可以按 git worktree 启动，每个更改实例独立运行
- 集成 Chrome DevTools Protocol，使代理能够捕获 DOM 快照、截图并导航
- 提供本地可观测性栈（日志、指标、追踪），代理可以查询 LogQL 和 PromQL

### 人类角色转变

在 ralph-wiggum-loop 中，人类工程师的角色从手动编写代码转变为：

- **设计系统**：定义项目结构、CI 配置、代码规则
- **指定意图**：编写高层次的任务提示，明确期望结果
- **构建反馈循环**：确保代理有足够的工具和上下文来自主工作

### 无手动编写代码哲学

团队采用“0 行手动编写代码”的约束，迫使工程团队专注于提升代理能力，而非直接修复代码。当代理失败时，团队会问：“缺少什么能力？如何让这些能力对代理可读且可强制执行？”

## 实施效果

根据团队报告，采用 ralph-wiggum-loop 后：

- 团队（最初 3 人，后扩展至 7 人）在五个月内合并了约 1,500 个 PR
- 每个工程师每天平均生成 3.5 个 PR
- 产品拥有内部日活用户和外部 alpha 测试者
- 构建速度估计是传统手动编码的 10 倍

## 适用场景

ralph-wiggum-loop 最适合以下场景：

- 需要高吞吐量的代码交付（如内部工具或快速原型）
- 有明确的代码规范和审查标准
- 团队愿意投资于代理基础设施和环境规范化
- 人类工程师时间稀缺，希望最大化杠杆效应

## 局限性

- 初期投入高：需要充分规范环境、创建代理技能和工具
- 代理可能无法处理高度不明确或创新的需求
- 在复杂系统或安全关键应用中，可能需要人类审查作为安全网
