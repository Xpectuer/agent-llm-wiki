# AGENTS.md 代理工作引导策略

> Source(s): raw/Harness engineering leveraging Codex in an agent-first world.md

**AGENTS.md**是一种以结构化 `AGENTS.md` 文件作为知识目录，指向更详细文档（如 `docs/`）的代理工作引导策略。该策略的核心是建立一个清晰、可执行的知识框架，让 AI 代理能够高效地理解代码库、项目规则和工作流程，从而自主完成开发任务。

## 背景与起源

AGENTS.md 的概念源于 OpenAI 在 2025 年发起的一项实验：**完全使用 Codex 代理构建一个软件产品，不写一行手动代码**。该团队在五个月内创建了约一百万行代码的软件产品，包含 1,500 个合并的 Pull Request，由小型工程团队（最初 3 人，后增至 7 人）驱动 Codex 完成。平均每位工程师每天产出 3.5 个 PR。

这一实验的核心哲学是：**“人类引导，代理执行”（Humans steer. Agents execute.）**。工程师的角色从编写代码转变为设计系统、明确意图并构建反馈循环，使 AI 代理能够可靠地工作。

## 核心工作原理

### 1. 知识目录化

`AGENTS.md` 文件位于仓库根目录，作为代理的“入口指南”。它包含：

- 项目结构概述
- 关键技术栈和工具链
- 编码规范和格式化规则
- 工作流程和审查要求
- 指向 `docs/` 目录下详细文档的链接

### 2. 代理驱动的开发流程

1. **任务描述**：工程师通过自然语言描述任务，运行代理。
2. **自主执行**：代理使用标准开发工具（`gh`、本地脚本等）获取上下文，生成代码、测试、CI 配置等。
3. **自审查循环**：代理调用自身或云端代理审查变更，形成拉尔夫·维格姆循环（Ralph Wiggum Loop），迭代直到所有代理审查者满意。
4. **人类监督**：人类可审查 PR，但不强制。大多数审查工作逐渐由代理间自动完成。

### 3. 增强环境可读性

为使代理更有效工作，团队优化了运行环境：

- **应用启动**：每个 git worktree 可独立启动应用，代理能在隔离环境中运行实例。
- **UI 可访问**：集成 Chrome DevTools 协议，代理可通过 DOM 快照、截图等操作 UI，复现并验证 bug。
- **可观测性**：每个 worktree 配备临时可观测性堆栈（日志、指标、追踪），代理可用 LogQL 和 PromQL 查询，实现“启动时间 < 800ms”或“关键路径耗时 < 2s”等精确任务描述。

## 关键经验

### 人类工程师的新角色

- **系统性思维**：不再纠结于代码细节，而是设计让代理高效工作的环境。
- **深度优先分解**：将大目标分解为小构建块（设计、代码、审查、测试等），逐步解锁复杂任务。
- **缺失能力识别**：当代理失败时，工程师询问“什么能力缺失？如何让它对代理既清晰又可执行？”

### 瓶颈转移

初始由代码生成速度慢，但后期主要瓶颈变为 **人类 QA 容量**。解决办法是让更多系统方面（UI、日志、指标）对代理直接可读，减少人工干预需求。

## 使用场景

AGENTS.md 特别适合以下场景：

- **AI 优先的软件开发**：团队希望最大化利用 AI 代理的自主编码能力。
- **大型代码库**：需要为代理提供清晰的结构和导航指引。
- **高吞吐量开发**：每天处理大量 PR，需要自动化审查流程。
- **持续集成/持续部署（CI/CD）**：自动生成 CI 配置、部署脚本等。

## 最佳实践

1. **保持 AGENTS.md 简洁**：只包含最关键的信息，详细内容放在 `docs/` 子目录。
2. **定期更新**：随着项目演进，同步更新代理知识库。
3. **可执行规则**：使用明确、可验证的规则（如“所有函数必须有文档字符串”），而非模糊指南。
4. **反馈循环**：建立代理自我审查机制，减少人类参与。
5. **环境隔离**：为每个任务创建隔离运行环境，避免冲突。
