# codex-cli

> Source(s): raw/Harness engineering leveraging Codex in an agent-first world.md

**codex-cli** 是 OpenAI Codex 的命令行界面工具，允许代理（包括 GPT‑5 等模型）直接生成和操作代码。该工具是 OpenAI 为构建「代理优先」的软件工程环境而开发的组件，其核心设计理念是：**人类负责设计、指定意图并构建反馈回路，而代码执行完全由代理完成**。

## 背景与实验

2025 年 8 月，OpenAI 团队启动了一项实验：从空白的 Git 仓库开始，使用 codex-cli 构建一款软件产品，并规定 **0 行手动编写的代码**。该产品拥有内部日活用户和外部 alpha 测试人员，能够正常上线、部署、出现故障并修复，唯一不同的是——每一行代码都由 Codex（GPT‑5）编写。

团队估计，使用 codex-cli 完成这个项目的时间大约仅为手动编写代码所需时间的 **1/10**。在五个月的时间里，仓库积累了约 **100 万行代码**，产生并合并了大约 **1500 个拉取请求**，而人力团队仅由 **3 名工程师** 驱动 Codex，后来扩展到 7 名工程师，吞吐量反而有所提升。

## 工作方式

codex-cli 的工作流程以提示（prompt）为核心：

1. 工程师描述任务，运行代理。
2. 代理使用标准开发工具（`gh`、本地脚本、仓库内嵌技能）直接获取上下文，无需人工复制粘贴。
3. 代理自动打开拉取请求（PR），并执行自审、请求其他代理审核（本地或云端）、根据反馈迭代，直到所有代理审核者满意（即 **Ralph Wiggum Loop**）。
4. 人工可以审核 PR，但并非必需。随着时间推移，大部分审核工作已转为代理间自动完成。

整个团队遵循的核心哲学是：**no manually-written code**（无手动编写代码）。

## 工程师角色的重新定义

放弃手动编码并未使工程师失业，而是引入了不同维度的工程工作，聚焦于**系统、框架和杠杆效应**：

- 早期进展较慢，并非因为 Codex 能力不足，而是因为环境**定义不足**——代理缺少所需的工具、抽象和内部结构来推进高级目标。
- 工程师的主要工作变成了**使代理能做有用的工作**：深度优先分解目标（设计、编码、审核、测试等），提示代理构建这些模块，并用它们解锁更复杂的任务。
- 当任务失败时，修复方式几乎从来不是「再试一次」，而是工程师介入并问：「缺失了什么能力？如何才能让它对代理既清晰又可强制执行？」

工程师的稀缺资源不再是写代码的能力，而是**人的时间和注意力**。

## 增强应用的可读性

随着代码吞吐量激增，瓶颈变成**人的 QA 容量**。为此，团队为代理添加了更多能力，使应用本身对 Codex 更加直接可读：

- **应用 UI 可读**：让应用在每个 Git 工作树（worktree）上可启动，Codex 可针对每个变更启动并驱动一个实例。通过将 Chrome DevTools 协议集成到代理运行时，创建了用于操作 DOM 快照、截图和导航的技能，使 Codex 能够复现 bug、验证修复、推理 UI 行为。
- **可观测性工具可读**：日志、指标和追踪通过本地可观测性栈暴露给 Codex，该栈对每个 worktree 是临时的。代理可使用 LogQL 查询日志、使用 PromQL 查询指标。这使得诸如「确保服务启动时间低于 800 毫秒」或「这四条关键用户旅程中没有任何 span 超过两秒」的提示变得可行。

团队经常看到单个 Codex 运行在一个任务上持续工作。

## 关键优势

- **显著缩短开发周期**：100 万行代码的项目仅用数月完成，人力仅为传统团队的很小一部分。
- **代理间增强协作**：代码审核、QA 测试等重复性工作可完全由代理对代理完成，解放人力聚焦于更高层次的设计与决策。
- **自动化程度高**：从初始脚手架（仓库结构、CI 配置、格式化规则、包管理器设置、应用框架）到 AGENTS.md 文件本身，均由 codex-cli 生成。
