# Managed Agents

> Source(s): raw/Scaling Managed Agents Decoupling the brain from the hands.md

**Managed Agents** 是 Anthropic 推出的托管服务，旨在通过将会话、控制循环和沙盒解耦，来运行长期代理任务。该服务的设计参考了操作系统中“进程”和“文件”等抽象概念，旨在为尚未设想的程序提供稳定的接口，使底层的实现细节可以自由更替而不影响上层逻辑。

## 设计理念：虚拟化与解耦

构建 Managed Agents 的核心在于解决计算领域的一个老问题：如何为“尚未设想的程序”设计系统。Managed Agents 借鉴了操作系统虚拟化硬件的经验（例如 `read()` 命令不关心底层是磁盘还是 SSD），将代理的组件进行了虚拟化处理。

该系统将代理架构中的三个核心组件进行了虚拟化解耦：

*   **Session (会话)**：记录所有发生事情的仅追加日志。
*   **Harness (控制循环)**：调用 Claude 并将 Claude 的工具调用路由到相关基础设施的循环。
*   **Sandbox (沙盒)**：执行环境，允许 Claude 在其中运行代码和编辑文件。

这种设计使得每个组件的实现都可以在不干扰其他组件的情况下进行替换。系统对这些接口的形态有明确的规范，但不关心其背后运行的具体实现。

## 架构演进

### 早期挑战：The "Pet" Problem

在最初的设计中，Managed Agents 将所有代理组件（会话、harness、沙盒）放置在单个容器中。虽然这种做法有文件编辑直接通过系统调用、无需设计服务边界的好处，但也带来了严重的架构问题，即所谓的 **“宠物”** 问题：

*   **耦合风险**：由于所有组件共享环境，服务器变成了难以丢失的“宠物”。一旦容器故障，会话即丢失；如果容器无响应，工程师必须像照顾生病的宠物一样尝试修复它。
*   **调试困难**：由于无法通过 WebSocket 事件流准确定位故障源头（是 harness bug、数据包丢失还是容器离线），且容器中包含用户数据，导致系统几乎无法进行有效调试。
*   **集成限制**：Harness 假定 Claude 的工作对象与本身共处于同一容器中。这导致客户若想连接 Claude 到其私有云，必须进行网络对等或在客户环境中运行 harness，限制了灵活性。

### 解决方案：将大脑与双手解耦

为了解决上述问题，Managed Agents 采用了将“大脑”与“双手”分离的架构：

*   **解耦定义**：
    *   **大脑**：Claude 及其 Harness。
    *   **双手**：执行动作的沙盒和工具。
    *   **会话**：会话事件日志。
    *   这三部分变成了互不做太多假设的接口，各自可以独立失败或替换。

*   **Harness 离开容器**：
    *   Harness 不再运行于容器内部，而是像调用其他工具一样通过 `execute(name, input) → string` 调用容器。
    *   容器变成了“牲畜”，如果容器死亡，Harness 会捕获工具调用错误并将其传递给 Claude。如果 Claude 决定重试，可以使用标准配方 `provision({resources})` 重新初始化一个新容器，而无需人工修复。

*   **Harness 故障恢复**：
    *   Harness 本身也变成了“牲畜”。
    *   由于会话日志位于 Harness 之外，Harness 在崩溃时无需保存任何状态。
    *   当 Harness 发生故障时，可以通过 `wake(sessionId)` 重启新的 Harness，并通过 `getSession(id)` 获取事件日志，从最后一个事件恢复执行。

## 动态适配性

Managed Agents 的设计也考虑到了模型能力的快速演进。传统的 Harness 往往基于模型“做不到”某些事情的假设来编写（例如 Claude Sonnet 4.5 曾表现出的“上下文焦虑”）。然而，随着模型升级（例如 Claude Opus 4.5），这些旧的限制性假设可能变得过时甚至成为累赘。

通过构建独立于具体实现之外的托管服务，Managed Agents 旨在提供一套能够经受住特定模型版本更迭的接口，从而实现长期的有效运行。
